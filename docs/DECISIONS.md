# 결정 기록

상태 표기는 `결정`, `제안`, `미정`을 사용한다. `제안`은 구현 전 검증으로 바뀔 수 있다.

## 현재 결정 사항

| ID | 상태 | 결정 | 근거/영향 |
|---|---|---|---|
| D-001 | 결정 | Expo SDK 54, React Native, TypeScript strict, Expo Router를 유지한다. | 현재 프로젝트 설정과 일치하며 SDK 변경은 작업 범위 밖이다. |
| D-002 | 결정 | iPhone을 우선 개발하고 Android 호환성을 유지한다. | 촬영 중심 초기 사용자 경험을 iOS에서 먼저 다듬되 플랫폼 종속 도메인 로직을 피한다. |
| D-003 | 결정 | OCR 결과는 자동 저장하지 않고 사용자가 수정·삭제·추가 후 확정한다. | 오인식이 학습 데이터로 굳어지는 위험을 줄인다. |
| D-004 | 결정 | 초기 데이터는 로컬 우선이며 계정과 Supabase 동기화는 MVP 이후다. | 오프라인 복원력과 초기 구현 범위를 우선한다. |
| D-005 | 결정 | 무료 이미지 검색을 우선하고 적합한 결과가 없을 때만 명시적으로 AI 이미지를 생성한다. | 비용과 불필요한 생성 사용을 줄이고 실제 이미지 선택권을 보장한다. |
| D-006 | 결정 | 검색 이미지에는 제공자·원본·작가·라이선스/출처 메타데이터를 저장한다. | 저작권 표시와 삭제 대응에 필요하다. |
| D-007 | 결정 | 간호·의학 콘텐츠는 학습 참고용이며 진단·처방 기능으로 제공하지 않는다. | 부정확한 정보의 안전 위험과 제품 범위를 명확히 한다. |
| D-008 | 결정 | 비밀 provider 키는 모바일 앱에 넣지 않고 서버 또는 Edge Function에서 관리한다. | 앱 번들과 `EXPO_PUBLIC_*` 값은 추출 가능하다. |

## 구현 방향 제안

| ID | 상태 | 제안 | 확정 조건 |
|---|---|---|---|
| D-009 | 제안 | 구조화 로컬 저장소로 SQLite, 사진에는 앱 전용 파일 저장소를 사용한다. | SDK 54 API와 migration/backup 동작 spike 통과 |
| D-010 | 제안 | 화면은 repository를 통해 로컬 데이터를 읽고 외부 결과도 먼저 로컬에 반영한다. | UI 갱신과 transaction prototype 검증 |
| D-011 | 제안 | 복습 평가는 `again/hard/good/easy` 네 단계와 버전이 있는 알고리즘을 사용한다. | 사용자 테스트와 초기 간격 정책 확정 |
| D-012 | 제안 | 로컬 ID는 UUID, 변경 동기화는 outbox와 tombstone을 사용한다. | Supabase schema 및 충돌 실험 |

## 결정 갱신 규칙

확정 시 해당 행을 `결정`으로 바꾸고 날짜, 선택지, 근거, 되돌림 비용을 기록한다. 데이터 모델·보안·로드맵에 영향이 있으면 같은 변경에서 관련 문서를 함께 수정한다. provider 이름이나 키 이름은 실제 계약이 정해지기 전에 확정된 것처럼 코드에 박지 않는다.
