# LinkVoca 작업 지침

이 저장소에서 작업하는 Codex 및 자동화 에이전트는 아래 절차를 따른다. 이 파일은 저장소 전체에 적용된다.

## 작업 절차

1. 작업 전에 이 파일, `README.md`, 관련 `docs/` 문서, `package.json`, `app.json`을 읽는다.
2. Expo 관련 구현 전에는 반드시 SDK 54의 정확한 버전 문서인 <https://docs.expo.dev/versions/v54.0.0/>를 확인한다. 최신 버전의 예제를 SDK 54에 그대로 적용하지 않는다.
3. 요청 범위와 현재 변경 상태를 확인하고, 사용자의 기존 변경을 덮어쓰지 않는다.
4. 기능은 `촬영/선택 → OCR → 사용자 교정 → 단어 정보/이미지 → 저장 → 복습` 흐름 중 어느 단계에 속하는지 먼저 정한다.
5. Expo Go에서 가능한 구현인지 확인한다. 사용자 정의 네이티브 모듈이 필요하면 임시 대체 경로와 development build 전환 조건을 문서화한다.
6. 최소 범위로 구현하고 lint, 타입 검사, 관련 테스트를 실행한다.
7. 기능·데이터·보안 결정이 바뀌면 같은 작업에서 관련 `docs/`도 갱신한다.

## TypeScript 규칙

- `strict` 모드를 유지하며 `any`, 무분별한 타입 단언, `@ts-ignore`를 사용하지 않는다. 불가피하면 이유와 제거 조건을 주석으로 남긴다.
- 외부 입력(OCR 응답, API 응답, 로컬 DB 행, 딥 링크)은 신뢰하지 않고 경계에서 검증한 뒤 도메인 타입으로 변환한다.
- 화면 컴포넌트에 OCR·DB·네트워크 로직을 직접 넣지 않는다. UI, use case/service, repository/provider 경계를 분리한다.
- 상태 값은 문자열 남용 대신 판별 가능한 union을 사용한다. 예: OCR 작업은 `queued | processing | needs_review | completed | failed`로 표현한다.
- 단어 ID, 작업 ID 등은 의미 있는 이름을 사용하고 nullable 값은 실제 부재가 허용되는 경우에만 쓴다.
- Expo Router 경로는 typed routes 설정을 유지하고, 화면 파라미터는 명시적으로 타입을 좁힌다.
- 파일·컴포넌트는 단일 책임을 우선하며, 공용 타입과 도메인 규칙은 화면 파일에서 분리한다.

## 수정 범위

- 요청과 직접 관련된 파일만 수정한다. 대규모 리팩터링, 패키지 교체, SDK 업그레이드를 끼워 넣지 않는다.
- `package.json`, lockfile, native 폴더, Expo SDK 버전 변경은 사용자의 명시적 요청 없이는 하지 않는다.
- iPhone을 우선 검증하되 Android의 권한, 뒤로 가기, edge-to-edge, 파일 URI 차이를 무시하지 않는다.
- 생성 파일, 이미지 원본, OCR 결과의 보존 기간과 삭제 동작을 함께 설계한다.

## 보안 및 데이터 원칙

- 교재 사진에는 이름, 학교, 메모 등 개인정보가 있을 수 있다. 원본 업로드를 기본값으로 가정하지 말고 최소 수집·로컬 우선·명시적 동의를 적용한다.
- 앱 번들에 포함되는 `EXPO_PUBLIC_*` 값은 비밀이 아니다. 비밀 API 키는 서버/Edge Function에만 둔다.
- 로그, fixture, 스크린샷, 커밋에 토큰·개인 사진·전체 OCR 원문을 넣지 않는다.
- 이미지에는 제공자, 원본 URL, 작가, 라이선스/출처 URL을 저장한다. 라이선스가 불명확한 이미지는 채택하지 않는다.
- 간호·의학 영어의 뜻과 예문은 학습 보조 정보이며 진단·처방 근거가 아님을 표시하고, 검증된 사전 출처와 사용자 교정을 우선한다.

## 테스트 및 검증

- 문서만 수정한 경우 링크, 용어, 파일 목록, `git diff --check`를 확인한다.
- 코드 수정 시 최소한 `npm run lint`와 `npx tsc --noEmit`을 실행한다.
- 데이터/복습 로직은 단위 테스트, repository는 통합 테스트, 촬영부터 카드 생성까지는 핵심 흐름 테스트를 추가한다.
- iOS 우선으로 실제 기기 권한 거부·취소·재시도를 확인하고, Android 호환성도 릴리스 전 확인한다.
- 실행하지 못한 검증은 성공한 것처럼 보고하지 않고 이유와 남은 위험을 적는다.

## 완료 보고 형식

완료 응답에는 다음을 간결하게 포함한다.

1. 결과: 사용자 관점에서 무엇이 달라졌는지
2. 변경 파일: 생성·수정한 파일 목록과 핵심 내용
3. 검증: 실행한 명령과 결과
4. 미결 사항: 알려진 제한, 결정이 필요한 항목, 실행하지 못한 검증

커밋, 배포, 외부 서비스 데이터 변경은 사용자가 명시적으로 요청한 경우에만 수행한다.
